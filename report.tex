\begin{center}
    \Large \textbf{Executive Summary}
\end{center}
\vspace{0.5cm}
\onehalfspacing

This report documents the design, development, and evaluation of a Sinhala-language multimodal conversational AI agent for automated food ordering, built for \textit{Wakwalle Kade} (\textit{\si{වක්වැල්ලේ කඩේ}}), a food shop near the University of Ruhuna campus in Hapugala, Wakwalle, Galle. The system enables customers to place food orders through WhatsApp messaging and telephone voice calls, conducting the entire ordering conversation in Sinhala, English, or a natural mixture of both languages.

The project applies four Advanced AI techniques: (1) \textbf{Natural Language Processing} for intent classification and entity extraction; (2) \textbf{Transformer-based fine-tuning} of the Gemma 4 E2B large language model using Low-Rank Adaptation (LoRA) on a synthetic dataset of 100 Sinhala food-ordering conversations; (3) \textbf{Prompt Engineering} with zero-shot, few-shot, and chain-of-thought strategies; and (4) \textbf{Retrieval-Augmented Generation (RAG)} using FAISS vector search for real-time menu and pricing retrieval.

Quantitative evaluation across three model configurations --- zero-shot baseline, few-shot prompt-engineered, and LoRA fine-tuned --- demonstrates measurable improvements at each stage. The few-shot prompt-engineered model achieves a Task Completion Rate of 62\% (vs.\ 38\% zero-shot) and the fine-tuned model achieves 67\%. The complete system is deployed on Lightning AI with an NVIDIA L4 GPU and handles end-to-end ordering conversations over real Twilio WhatsApp and voice channels.

\chapter{Problem Statement and Motivation}

\section{Background}
The rapid adoption of smartphones and messaging applications across South Asia has opened new opportunities for automating service interactions in local languages. In Sri Lanka, Sinhala is spoken by approximately 75\% of the population \cite{ethnologue2024}, yet the vast majority of conversational AI systems and automated ordering platforms operate exclusively in English. This language barrier excludes a large segment of the population from AI-driven services, particularly in rural and semi-urban communities.

Food ordering is a high-frequency, structured conversational task well-suited to automation. A customer must specify a menu item, clarify a variant, provide a quantity, and confirm the order --- a repeatable dialogue pattern that a well-designed agent can handle reliably. When this interaction occurs in the customer's native Sinhala over a familiar channel such as WhatsApp or a phone call, the technology becomes genuinely accessible.

\section{Problem Statement}
Small food shops near Sri Lankan university campuses rely on phone calls and walk-in orders. Phone-based ordering requires staff availability around the clock, leading to missed calls during peak hours and errors caused by miscommunication. Existing automated chatbot solutions do not support Sinhala and are not tailored to local shop menus, pricing structures, or ordering customs.

The specific problem addressed is: \textit{How can a low-cost, deployable conversational AI agent handle Sinhala food ordering conversations over both voice and WhatsApp with sufficient accuracy and naturalness for practical use at a local food shop?}

\section{Research Questions}
\begin{enumerate}
    \item How effectively can a fine-tuned Sinhala LLM handle food-ordering conversations?
    \item Does fine-tuning improve performance compared to prompt-only approaches?
    \item How accurately can the system identify food-ordering intents and extract entities?
    \item Can a Sinhala conversational AI be deployed on accessible cloud GPU infrastructure with acceptable latency?
\end{enumerate}

\section{Motivation and Real-World Relevance}
The motivation for this project is threefold. First, it addresses a genuine gap: no Sinhala-language food ordering AI exists. Second, it demonstrates that advanced AI techniques --- LLM fine-tuning, RAG, and prompt engineering --- can be applied to low-resource languages with a modest dataset. Third, the system is practically deployable and has been demonstrated live on real Twilio channels.

\section{Objectives}
\begin{itemize}
    \item Build a Sinhala conversational AI agent capable of complete food ordering dialogues over WhatsApp and voice.
    \item Fine-tune Gemma 4 E2B on a synthetic Sinhala dataset using LoRA for domain adaptation.
    \item Implement intent recognition, entity extraction, and dialogue state management using NLP.
    \item Integrate a FAISS-based RAG pipeline for accurate menu retrieval.
    \item Evaluate and compare zero-shot, few-shot, and fine-tuned model configurations.
    \item Deploy the complete system on cloud GPU infrastructure with public webhook access.
\end{itemize}

\section{Scope and Limitations}

\subsection{Scope}
\begin{itemize}
    \item \textbf{Shop:} Wakwalle Kade only (one fixed menu of eight items, pickup only).
    \item \textbf{Channels:} WhatsApp (text) and telephone voice calls.
    \item \textbf{Languages:} Sinhala, English, and Sinhala-English mixed.
    \item \textbf{Model:} Google Gemma 4 E2B fine-tuned with LoRA via Unsloth on Lightning AI L4 GPU.
\end{itemize}

\subsection{Limitations}
\begin{itemize}
    \item \textbf{STT accuracy:} Twilio's Sinhala STT occasionally misrecognises digits spoken in Sinhala.
    \item \textbf{Dataset size:} The LoRA adapter was trained on 100 synthetic conversations; a larger real-world dataset would improve generalisation.
    \item \textbf{Session persistence:} Sessions are stored in memory and expire after 30 minutes; server restarts clear active sessions.
    \item \textbf{Single shop:} Multi-restaurant support would require per-restaurant RAG indices.
\end{itemize}

\chapter{Literature Review}

\section{Conversational AI and Task-Oriented Dialogue Systems}
Task-oriented dialogue systems have evolved from hand-crafted finite state machines through statistical dialogue managers to end-to-end neural approaches \cite{young2013}. Modern LLM-based systems such as those built on GPT-4 and Gemma have demonstrated the ability to manage complex multi-turn dialogues through in-context learning, without requiring explicit state machine design \cite{brown2020}.

\section{Low-Resource Language NLP}
Sinhala is classified as a low-resource language due to the scarcity of large-scale labelled corpora, pre-trained models, and benchmarks \cite{joshi2020}. Multilingual pre-trained models such as mBERT and XLM-RoBERTa have shown reasonable zero-shot transfer to Sinhala, though performance degrades significantly on domain-specific vocabulary and conversational contexts \cite{conneau2020}.

\section{Transformer-Based Models and Fine-Tuning}
Large transformer-based language models pre-trained on massive corpora have established state-of-the-art performance across a wide range of NLP benchmarks \cite{vaswani2017}. LoRA injects trainable low-rank matrices into the attention layers of a transformer, leaving the original weights frozen, and matches full fine-tuning performance with fewer than 1\% of trainable parameters \cite{hu2022}. QLoRA extended this to 4-bit quantized models, enabling fine-tuning on consumer GPUs \cite{dettmers2023}. Google's Gemma 4 family \cite{team2024} demonstrates competitive multilingual performance and is the base model used in this project.

\section{Prompt Engineering}
Brown et al.\ \cite{brown2020} demonstrated that few-shot in-context learning achieves competitive performance across diverse tasks without gradient updates. Chain-of-thought prompting \cite{wei2022} further improves reasoning by eliciting intermediate steps. For conversational agents, system prompt design is critical: the prompt establishes persona, constraints, and example behaviours that the model follows throughout a dialogue.

\section{Retrieval-Augmented Generation (RAG)}
RAG frameworks address the hallucination problem in LLMs by conditioning generation on retrieved documents \cite{lewis2020}. A query is encoded into a dense vector and matched against a pre-built index using FAISS \cite{johnson2021}. The top-$k$ retrieved passages are prepended to the generation prompt, grounding the response in factual content. Sentence-Transformers \cite{reimers2019} provide compact, high-quality embeddings suitable for small-scale menu retrieval.

\section{Voice Interfaces for Sinhala}
Twilio's Programmable Voice API provides managed STT supporting Sinhala (\texttt{si-LK}) via Google Cloud Speech-to-Text \cite{twilio2024}, eliminating the need for a custom STT pipeline. For Sinhala TTS synthesis, Google Gemini TTS API supports multilingual synthesis with automatic language detection \cite{google2024}.

\section{Research Gap}
Three specific gaps motivate this project:
\begin{itemize}
    \item No Sinhala task-oriented dialogue system exists for food ordering or any customer-service domain.
    \item No multimodal local-language ordering platform supports Sinhala across both voice and chat channels.
    \item The relative benefit of prompt engineering vs.\ fine-tuning for Sinhala domain tasks has not been quantitatively studied.
\end{itemize}

\section{Dataset Generation}
A synthetic dataset of 100 Sinhala food-ordering conversations was generated for LoRA fine-tuning, covering four language styles (25 conversations each): formal Sinhala, informal Sinhala, Sinhala-English mixed, and English with Sinhala confirmations. Each conversation covers at least five of the nine intent classes and follows the complete five-step ordering flow: item $\rightarrow$ variant $\rightarrow$ quantity $\rightarrow$ summary $\rightarrow$ confirmation.

\subsection{Data Format}
Conversations are stored in ShareGPT format and converted to the Gemma 4 chat template. A preprocessing step merges the system message into the first user turn, required by Gemma 4's strict user-assistant alternation constraint.

\section{NLP Pipeline}

\subsection{Intent Classification}
Intent is classified by the LLM in-context using nine classes: \texttt{food\_order}, \texttt{menu\_inquiry}, \texttt{price\_inquiry}, \texttt{protein\_select}, \texttt{portion\_select}, \texttt{package\_select}, \texttt{quantity\_change}, \texttt{confirm\_order}, and \texttt{cancel\_order}.

\subsection{Entity Extraction}
Key entities extracted per turn: item name, protein/portion/package variant, quantity, and phone number. Phone number is extracted by regex (\texttt{(\textbackslash+94|0)?[0-9]\{9,10\}|07[0-9]\{8\}}) or captured directly from Twilio's \texttt{From} field for voice calls.

\subsection{Text Postprocessing}
Generate output is stripped of Gemma 4 artefacts: \texttt{<end\_of\_turn>} tokens, \texttt{**Note:**} prefixes, and thinking-mode preambles.

\section{LLM Fine-Tuning}
LoRA fine-tuning was performed on \texttt{google/gemma-4-e2b-it} using Unsloth's \texttt{FastLanguageModel} on an NVIDIA L4 GPU.

\begin{itemize}
    \item Rank $r=16$, alpha 16, dropout 0.05; target: q/k/v/o and FFN projections.
    \item Trainable parameters: $\sim$40M (0.79\% of total).
    \item Batch size 1, gradient accumulation 8, cosine LR $2\times10^{-4}$, 3 epochs, 36 steps; final loss 0.8598.
\end{itemize}

\section{Prompt Engineering}

\subsection{Zero-Shot}
Model receives a brief role description and the menu, with no examples. Used as the baseline (M1).

\subsection{Few-Shot}
Four complete example conversations are embedded in the system prompt, covering all major order types. Used in production (M2).

\subsection{Chain-of-Thought}
A reasoning preamble instructs the model to identify intent before responding. Evaluated for ambiguous inputs.

\subsection{Production Prompt Design}
Three engineering decisions were critical:
\begin{enumerate}
    \item A \texttt{[INSTRUCTIONS --- do NOT acknowledge]} prefix prevents instruction paraphrasing.
    \item A priming synthetic assistant greeting places the model in an in-progress ordering context.
    \item Phone number injection into the system prompt when already known eliminates re-asking.
\end{enumerate}

\section{RAG Pipeline}
At startup, eight menu items are encoded using \texttt{all-MiniLM-L6-v2} (384-dim embeddings) and indexed in FAISS \texttt{IndexFlatL2}. On each user message, the top-4 nearest neighbours are retrieved and injected into the system prompt.

\section{Ordering Dialogue Flow}
\begin{enumerate}
    \item Customer states an item.
    \item Agent asks the appropriate clarifying question (protein / half-full / package size).
    \item Agent asks quantity.
    \item Agent shows full order summary with total and requests confirmation.
    \item On confirmation, order saved to \texttt{data/orders.csv} with unique order ID (e.g., \texttt{\#WK12345}).
\end{enumerate}

\section{WhatsApp Channel}
Messages arrive at \texttt{POST /whatsapp/incoming}. The sender's phone number is extracted from Twilio's \texttt{From} field and passed directly to the agent.

\begin{lstlisting}[language=Python, caption=WhatsApp webhook handler (app/main.py)]
@app.post("/whatsapp/incoming")
async def whatsapp_incoming(From: str = Form(...), Body: str = Form(...)):
    user_id = From.replace("whatsapp:", "").strip()
    agent = get_agent()
    reply = agent.respond(user_id=user_id, user_message=Body,
                          channel="whatsapp", phone_number=user_id)
    twiml = MessagingResponse()
    twiml.message(reply)
    return Response(content=str(twiml), media_type="application/xml")
\end{lstlisting}

\section{Voice Channel}
\begin{itemize}
    \item \texttt{/voice/incoming} --- synthesises a Sinhala greeting and returns TwiML \texttt{<Gather language="si-LK">}.
    \item \texttt{/voice/process} --- receives STT transcript, generates agent reply, synthesises MP3, plays back. On confirmed order, returns \texttt{<Hangup>}.
\end{itemize}

\begin{lstlisting}[language=Python, caption=Voice confirmation and hangup (app/main.py)]
session = agent.get_session(user_id)
if session and session.confirmed:
    resp.play(reply_url)
    agent.reset_session(user_id)
    resp.hangup()
    return Response(content=str(resp), media_type="application/xml")
\end{lstlisting}

\section{TTS Implementation}
Sinhala TTS uses the Gemini REST API (\texttt{gemini-3.1-flash-tts-preview}). The API returns raw PCM (24 kHz mono), wrapped into WAV then converted to MP3. Files are cached by MD5 hash.

\section{Key Findings}
\begin{itemize}
    \item Few-shot prompt engineering provides the largest single improvement (TCR 38\% $\rightarrow$ 62\%) at zero additional training cost.
    \item LoRA fine-tuning on 100 examples yields a further measurable improvement (TCR +5 pp, KMS +0.05).
    \item The fine-tuned model required the few-shot system prompt; without it, repetitive outputs degraded quality.
    \item Production uses M2 (few-shot base) for greater stability in long conversations.
\end{itemize}

\chapter{Discussion}

\section{Interpretation of Results}
The results confirm that prompt engineering provides the majority of the performance gain. Gemma 4 already understands Sinhala natively --- the zero-shot 38\% TCR reflects the absence of domain-specific ordering flow guidance, not a language comprehension failure. Few-shot examples teach the shop-specific flow, which is absent from pre-training data.

The modest fine-tuning gain (5 pp TCR) is expected with only 100 training examples. The synthetic conversations cover the ordering flow but lack the diversity of real customer language. A dataset of 1,000 or more real conversations would likely make fine-tuning dominant.

\section{Challenges Encountered}
\begin{enumerate}
    \item \textbf{Gemma 4 role format:} Strict user-assistant alternation required merging system instructions into the first user turn and adding a synthetic priming assistant turn to prevent instruction-acknowledgement behaviour.
    \item \textbf{Sinhala TTS:} Multiple Gemini TTS model versions failed for Sinhala; \texttt{gemini-3.1-flash-tts-preview} with raw PCM output was the working solution.
    \item \textbf{Voice STT phone number:} Sinhala number words were misrecognised by Twilio STT. Fixed by using caller \texttt{From} ID directly.
    \item \textbf{Call non-termination:} Voice agent looped after confirmation. Fixed by detecting \texttt{session.confirmed} and returning \texttt{<Hangup>} immediately.
    \item \textbf{SIP trunk blocking:} An outbound SIP trunk attached to the Twilio number blocked inbound calls; removed via the Twilio REST API.
    \item \textbf{CUDA OOM during fine-tuning:} Resolved with batch size 1, gradient accumulation 8, and \texttt{PYTORCH\_CUDA\_ALLOC\_CONF=expandable\_segments:True}.
\end{enumerate}

\section{Lessons Learned}
\begin{itemize}
    \item A well-crafted few-shot prompt with a capable multilingual base model can outperform a fine-tuned model trained on a small synthetic dataset.
    \item Infrastructure challenges (SIP trunks, TTS compatibility, STT for phone numbers) take as much time as model development.
    \item Capturing channel metadata at the infrastructure layer eliminates unreliable in-conversation collection (e.g., spoken phone numbers via STT).
    \item Synthetic datasets are a valid starting point but are no substitute for real conversational data.
\end{itemize}

\section{Ethical Considerations}
\begin{itemize}
    \item Phone numbers captured via Twilio are used only for order identification and not shared with third parties.
    \item The system does not store voice recordings; only STT transcripts are processed.
    \item The synthetic dataset was generated without using real customer data.
\end{itemize}

\chapter{Conclusion and Future Work}

\section{Conclusion}
This project delivered a live, end-to-end Sinhala conversational AI food ordering system for Wakwalle Kade. The system handles complete ordering dialogues over WhatsApp and telephone in Sinhala, English, and code-switched language. Four Advanced AI techniques were integrated: NLP intent and entity extraction, LoRA fine-tuning, few-shot prompt engineering, and FAISS-based RAG.

Quantitative evaluation demonstrated an 81\% relative improvement in keyword match score from prompt engineering alone, and a further 8.6\% improvement from LoRA fine-tuning on 100 domain-specific synthetic examples. The deployed system successfully confirmed real orders in live end-to-end tests on both channels.

\section{Key Contributions}
\begin{itemize}
    \item A live WhatsApp and voice ordering agent deployed on cloud GPU infrastructure.
    \item A 100-conversation synthetic Sinhala food-ordering dataset (\texttt{data/synthetic\_dataset.json}).
    \item A LoRA-fine-tuned Gemma 4 E2B adapter published at \texttt{udithanuwanw/gemma4-sinhala-food-ordering}.
    \item A framework-free FAISS RAG pipeline for menu retrieval.
    \item Automatic CSV order logging with unique order IDs.
\end{itemize}

\section{Future Work}
\begin{itemize}
    \item \textbf{Larger real dataset:} Real WhatsApp conversation exports would substantially improve fine-tuning quality.
    \item \textbf{Multi-restaurant support:} Per-restaurant RAG indices with a single agent infrastructure.
    \item \textbf{Persistent sessions:} Redis-backed session storage to survive server restarts.
    \item \textbf{STT post-processing:} Sinhala digit normalisation layer for common STT misrecognitions.
    \item \textbf{Order dashboard:} Real-time web interface for the shop owner to manage incoming orders.
    \item \textbf{RLHF:} Using real customer feedback to improve conversational quality through reinforcement learning from human feedback.
\end{itemize}

\section{FAISS RAG Index Construction}
\begin{lstlisting}[language=Python, caption=RAG index construction at startup (app/rag.py)]
from sentence_transformers import SentenceTransformer
import faiss, numpy as np

class MenuRAG:
    def __init__(self, menu_path):
        self._encoder = SentenceTransformer("all-MiniLM-L6-v2")
        items = json.loads(menu_path.read_text())
        texts = [self._item_to_text(i) for i in items]
        vecs = self._encoder.encode(texts).astype(np.float32)
        self._index = faiss.IndexFlatL2(vecs.shape[1])
        self._index.add(vecs)
        self._texts = texts

    def get_context_for_query(self, query, top_k=4):
        vec = self._encoder.encode([query]).astype(np.float32)
        _, idx = self._index.search(vec, top_k)
        return "\n".join(self._texts[i] for i in idx[0])
\end{lstlisting}
